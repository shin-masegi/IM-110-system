# /sync-protocol

`docs/protocol-rs232c.md` の定義と、本体 (`../IM-110/`) ・プローブ (`../IM-110_Probe/`)
両実装との食い違いを検出します。本プロトコルは **ASCII テキスト・行指向**（バイナリフレームではない）。

## 前提

- セッション内で `/add-dir ../IM-110` と `/add-dir ../IM-110_Probe` が済んでいること
- 済んでいなければ実行前にユーザーに促すこと
- ソース配置: 両リポとも `Core/Src/` (ソース) と `Core/Inc/` (ヘッダ)

## 手順

### 1. 仕様の抽出 (`docs/protocol-rs232c.md`)

以下を読み取る:
- §3.1 コマンド一覧表: ASCII コマンド名 (例: `VR`, `SP`, `SEL`...)、引数書式、応答種別、プローブ側実装の参照先 (`IM_110.c:行番号`)
- §3.2 応答詳細: `VR`, `MD`, `RPP`, `WPP`, `RPI` の文字列フォーマット
- §3.3 自律送信: `MS` による周期 1000 ms の測定値送信
- §4 タイミング: ボーレート、`MS` 周期 (1000 ms)、`HAL_UART_Transmit` タイムアウト (100 ms)、応答タイムアウト (実装無し)
- §5 エラー処理: `NG\r\n` の発生条件、CRC 無し、通信断検出無し
- §6 起動シーケンス: 本体は `MS\r\n` 送信のみ、VR ハンドシェイク無し

### 2. プローブ側実装の検証 (`../IM-110_Probe/Core/Src/IM_110.c`)

- `CheckSerialCMD()` (行 313 付近) 内の `strncmp(...)` ディスパッチ列挙を抽出。§3.1 の 14 コマンドが全て含まれるか、順序も含めて比較
- 各コマンドの引数パース (`strtok`, `strtof`, `strtol`, `strtoul`) と範囲比較 (`if (x < MIN || x > MAX)` 等) が §3 の範囲と一致するか
- 応答送信の `sprintf` 書式文字列が §3.2 と一致するか（特に `VR` の `"%s Ver.%d.%02d\r\n"`、`MD` の `"%f, %f, %f, %f, %f, %f\r\n"`）
- `TIMER_UART_OUTPUT` 定数 (`../IM-110_Probe/Core/Inc/IM_110.h`) と §4 の 1000 ms が一致するか
- 受信バッファサイズ (`RecvData[512]` in `IIJIMA_Templete.c`) と §2.4 の 510 バイト制限が整合するか

### 3. 本体側実装の検証 (`../IM-110/Core/Src/` 配下)

- `Probe_SendSerialData(..., "<CMD>\r\n")` で送信されている全リテラル文字列を列挙（`IM_110.c`, `Wire.c`, `WireSub.c`, `LinkSerial.c` を grep）
- 列挙されたコマンドが §3.1 に全て定義されているか
- 応答パーサ（例: `Probe_Process_MD`, `Probe_Process_VR` 等の関数 — `IM_110.c` 内）の `strtof` / `strncmp` が §3.2 のフォーマットに整合するか
- `HAL_UART_Transmit(&hlpuart1, ..., TIMEOUT)` の TIMEOUT 値が §4 の 100 ms と一致するか
- 起動シーケンス (`Probe_uart_init` → `Probe_Request_MD`) が §6 と一致するか

### 4. クロスチェック観点

| チェック項目 | 不一致時の扱い |
|---|---|
| 仕様のコマンドがプローブ `CheckSerialCMD` に実装されているか | **エラー** (プローブが受理できない) |
| 本体が送信しているコマンドが仕様に定義されているか | **エラー** (未定義コマンド) |
| 仕様のコマンドが本体から送信されているか | 情報 (未使用 or 将来用) |
| 応答フォーマットが仕様とプローブ送信で一致するか | **エラー** |
| 応答フォーマットが仕様と本体パーサで一致するか | **エラー** |
| パラメータ範囲 (例: `SP` の 0.0〜1.0) が仕様とプローブ実装で一致するか | 警告 |
| 自律送信周期 (`TIMER_UART_OUTPUT`) が §4 と一致するか | 警告 |
| 送信タイムアウト (`HAL_UART_Transmit` 第 4 引数) が §4 と一致するか | 警告 |
| 行終端が両側とも `\r\n` で揃っているか | 警告 (片側が LF のみでも実動作するが仕様と乖離) |

### 5. レポート出力

以下フォーマットで markdown 出力:

```markdown
# Protocol Sync Report (YYYY-MM-DD)

## Summary
- 仕様コマンド数: 14
- プローブ受理: N / 14
- 本体送信 (発見分): M / 14 (残りは spec のみ定義で未使用)
- 不整合 (Error): X
- 警告 (Warning): Y

## Errors
(各エラーの該当ファイル:行と差分を記載)

## Warnings
- [W1] `MS` 自律送信周期: 仕様 1000ms / プローブ実装 `TIMER_UART_OUTPUT = 1000` ✓ / 本体側監視タイムアウトは §9 TODO のまま未実装
...

## Informational
- 本体が送信していない仕様コマンド: `SP`, `SEL`, `SS`, `SID`, `SADZ`, `SADS`, `SADA`, `RPP`, `WPP`, `RPI` など
  （これらは調整用コマンド。現状 UI から直接送る経路は無い可能性 — 要確認）
```

## 重要な注意

- 実装を勝手に書き換えないこと。レポート出力のみ。
- ユーザーから明示的に「修正して」と言われた場合のみ、差分提案モードに入る。
- サブリポジトリが add-dir されていない場合は、該当側のチェックをスキップし、レポートにその旨を明記する。
- コマンド名は大文字前提（プローブ `strncmp` は大小区別）。実装が小文字を受理している場合は警告対象。
- `MS` は引数なしのトグル式（プローブ実装）だが、本体送信側で `MS,0` / `MS,1` を明示している場合は仕様との乖離として警告。
