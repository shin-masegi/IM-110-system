# IM-110 システム横断コンテキスト

このリポジトリは IM-110 製品全体を横断的に管理するための **調整用リポジトリ** です。
実際の本体・プローブのソースコードや回路資料は別リポジトリにあり、
このリポジトリは両者の整合性を保つための **仕様・ログ・ルール置き場** として機能します。

---

## 1. システム全体像

IM-110 は 2 基板構成の計測器です。

| 基板 | 役割 | リポジトリ | MCU |
|---|---|---|---|
| IM-110 本体基板 (IM-110T) | 操作・表示・通信・ログ | `../IM-110/` | STM32L452RET6 |
| IM-110 プローブ基板 (ID-160T) | O2・温度センサーのアナログ取得 | `../IM-110_Probe/` | STM32L452RET6 |

両者は **RS-232C ケーブルで接続**され、本体側がマスターとしてプローブを制御します。
物理レイヤの変換は本体側の MAX3232 (U16) が担当（プローブ側は TTL レベルで受ける想定、物理接続は要確認）。

---

## 2. リポジトリ構成（ローカル）

Claude Code はこの `IM-110-system/` ディレクトリから起動してください。
その後 `/add-dir ../IM-110` と `/add-dir ../IM-110_Probe` で両方のコードツリーをセッションに加えます。

```
~/dev/                         ← Claude Code の起動は IM-110-system から
├── IM-110/                    ← 本体リポジトリ（別GitHubリポ、独立バージョン管理）
├── IM-110_Probe/              ← プローブリポジトリ（別GitHubリポ、独立バージョン管理）
└── IM-110-system/             ← ★このリポ
    ├── CLAUDE.md              ← 横断コンテキスト（本ファイル）
    ├── docs/
    │   ├── protocol-rs232c.md ← 通信仕様の唯一の真実源
    │   └── debug-log/         ← Claude.ai からのデバッグ記録インボックス
    └── .claude/commands/      ← カスタムスラッシュコマンド
```

## 3. 絶対ルール

以下は例外なく守ってください。違反しそうな操作の前には必ずユーザーに確認すること。

### 3.1 プロトコル仕様の単一真実源
- RS-232C 通信のコマンド・応答・タイムアウトの定義は `docs/protocol-rs232c.md` のみが真実源。
- `IM-110/` または `IM-110_Probe/` 内のコードで通信処理を修正する場合、
  まず `docs/protocol-rs232c.md` を読んで整合を確認する。
- 仕様そのものを変える場合は `docs/protocol-rs232c.md` を先に更新し、
  その差分に従って両側のコードを修正する（順序が逆になってはいけない）。

### 3.2 両側同時修正のルール
通信に関わる変更は **必ず本体・プローブ両側のコードを同じセッションで見る**。
片側だけ修正してコミットしないこと。
変更対象の典型例:
- UART 送受信ハンドラ（本体側 = LPUART1、プローブ側 = USART2）
- コマンドパーサ / ディスパッチャ
- タイムアウト・リトライ定数
- パケットのバイトレイアウト構造体

### 3.3 バージョン管理の独立性
- 本体とプローブの GitHub リポジトリは独立。バージョン番号も独立。
- 片側だけ Ver. UP するのは自由だが、その際必ず:
  1. `docs/protocol-rs232c.md` のプロトコルバージョンを据え置きか上げるか判定
  2. 反対側の互換性を確認（後方互換を切る場合は反対側も同時リリース計画を立てる）
  3. このリポジトリの `docs/debug-log/` または `docs/release-notes.md` に「本体 vX × プローブ vY で検証」の組み合わせ履歴を残す

### 3.4 デバッグログのインボックス方式
Claude.ai で行った回路デバッグの議論は `docs/debug-log/` にコピペで保存される。
Claude Code はこのインボックスを読み、関連する `CLAUDE.md` や `protocol-rs232c.md`
に反映する役割を持つ（`/ingest-debug-log` コマンド参照）。

### 3.5 ハードウェア仕様の真実源
基板レベルの事実（MCU 型番・ピン配置・UART ch 割当・アナログ回路接続・電源構成など）は、
各サブリポ `Materials/` 配下の **回路図 PDF + ネットリスト (`.NET`)** を唯一の真実源とする。

- FW ソース（`.ioc` / `main.c` / `Makefile` / HAL 初期化コード）と回路図・NET の記述が食い違った場合、
  **まず回路図側を信じ、FW 側の誤設定を疑う**（FW は人手で CubeMX 設定しているため、基板リビジョンと
  ずれている可能性が常にある）。
- 回路図と NET の間で不一致がある場合はリビジョン差異の疑いがあるため、ユーザーに確認する。
- `CLAUDE.md` §4 / `protocol-rs232c.md` の物理層記述・その他ハード記載を更新する際は、
  必ず回路図 PDF + `.NET` と突き合わせ、根拠として参照ファイル名（と可能ならネット名）を残す。
- `Materials/` に回路図・NET が無い側については、確信を持った更新はしない（ユーザーに資料を要求する）。

#### 3.5.1 `Materials/` のリビジョン運用

- **`Materials/` 直下には現行リビジョンのみを置く**。回路図 PDF / ネットリスト (`.NET`) / 部品表 (BOM) は、
  すべて同一リビジョン（同じ基板実体を表すペア）で揃える。片方だけ更新されて不一致にならないよう注意。
- **旧リビジョンは `Materials/_archive/` に退避する**。新リビジョンに差し替える際、既存の直下ファイルを
  `Materials/_archive/<rev または日付>/` サブディレクトリに移してから、新しいファイルを直下に配置する。
  削除はしない（フィールド実機が旧リビジョンで稼働している可能性があるため、サポート時に必要）。
- **Claude Code は `_archive/` 配下を読まない**のが原則。§4 や `protocol-rs232c.md` の検証・更新、
  `/sync-protocol` や `/ingest-debug-log` 実行時、`_archive/` 配下は真実源として参照しない。
  ただしユーザーが明示的に過去リビジョンとの比較や互換性調査を指示した場合はこの限りではない。
- **ファイル名には基板名とリビジョン（または日付）を含める**。例: `回路図_IM-110T_プローブ基板_20260310.pdf`、
  `ネットリスト_IM-110T_プローブ基板_20260310.NET`。これにより `_archive/` に退避したあとも識別可能。
- `Materials/` 直下のファイル一覧と `_archive/` の最新サブディレクトリは、§4 を更新したときに根拠として
  CLAUDE.md 側に明記する。

## 4. ハードウェア要点（横断的に知っておくべきこと）

各項目の根拠は §3.5 のルールに従い、各サブリポ `Materials/` 配下の **回路図 PDF + ネットリスト (`.NET`)** を一次資料として検証済み。以下では必要に応じて NET 内のコンポーネント ID / ネット名を併記する。

### 本体基板 (IM-110T 本体基板)

- 現行リビジョン資料:
  - 回路図: `IM-110/Materials/未実装部品追加_回路図_IM-110T_本体基板_20260319.pdf`
  - ネットリスト: `IM-110/Materials/ネットリスト_IM-110T_本体基板.NET`
  - 部品表: `IM-110/Materials/未実装部品追加_部品表_IM-110T_本体基板20260421.xlsx`
- MCU: **STM32L452RET6** (LQFP64, Cortex-M4, 80MHz max, Flash 512kB / SRAM 160kB)  — NET: `U4 STM32L452R_64`
- 電源構成:
  - メイン: `V_BAT` → Q4 (P-MOSFET 主電源スイッチ, SW3 連動) → **MAX8881 (U9)** → `3V3` (3.3V)
  - 5V 系: `V_BAT` → **LTC3221 (U13)** チャージポンプ → `5V`、`CN4.1` 経由でプローブへ給電
- 本体⇔プローブ通信 (RS-232C):
  - STM32 LPUART1 (U4.8=RX / U4.9=TX) → **MAX3232 (U16)** → CN4 のケーブルへ
  - NET: `LPUART1_RXD U4.8/U16.12`, `LPUART1_TXD U4.9/U16.11`, `PROBE_RXD U16.14/CN4.3`, `PROBE_TXD U16.13/CN4.4`
- BLE 通信:
  - STM32 USART1 (U4.42=TX / U4.43=RX) → **74_2G125 バッファ (U7)** → **RN4871 (U5)**
  - NET: `UART1_TXD U4.42`, `UART1_RXD U4.43`, `BLE_TXD U5.8/U7.5`, `BLE_RXD U5.7/U7.6`
- ストレージ / RTC: SPI2 共有バス (SCK=U4.34, MOSI=U4.36, MISO=U4.35)
  - **R2045S RTC (U8)** — CS=U4.37 (`PC6_CS_RTC`)
  - **M95256 EEPROM (U11)** — CS=U4.39 (`PC8_NCS_MEM`)
- 表示: SPI1 (SCK=U4.21, MOSI=U4.23, CS=U4.38) → **CN3 (FH12-10S, 10pin FPC)** → LCD。実際の表示モジュールは Sharp メモリ液晶 **LS027B7** (`Materials/LS027B7.pdf` 参照)
- 操作スイッチ: SW1 (PC10, U4.51) / SW2 (PC11, U4.52)。電源スイッチ SW3 は `V_BAT` → Q4 のゲート制御系
- 外部コネクタ:
  - **CN1 (CON8)**: SWD/JTAG (JT_CK/DI/DO/MS, NJTRST, NRST)
  - **CN2 (CON6)**: USART1 TXD/RXD/CTS/RTS と NTEST を R13/R15/R16/R17/R18 経由で外部引き出し（デバッグ/テスト用）
  - **CN4 (SM05B-SHLS-TF 5pin)**: プローブケーブル用 — 5V (`CN4.1`), GND (`CN4.2`/`CN4.5`), PROBE_RXD (`CN4.3`), PROBE_TXD (`CN4.4`)

### プローブ基板 (IM-110T プローブ基板)

> 旧基板名義 `ID-160T REV8.1` / `IM_PROBE22` は retired。現行は **IM-110T プローブ基板**。

- 現行リビジョン資料:
  - 回路図: `IM-110_Probe/Materials/回路図_IM-110T_プローブ基板_20260310.pdf`
  - ネットリスト: `IM-110_Probe/Materials/ネットリスト_IM-110T_プローブ基板_20260310.NET`
  - 部品表: `IM-110_Probe/Materials/【部品表260309】IM_PROBE22_Rev.4.xls`
- MCU: **STM32G070KBT6** (LQFP32, Cortex-M0+, 64MHz max, Flash 128kB / SRAM 36kB) — NET: `IC20 STM32G070KBT6`
  - **本体 (STM32L452RE / Cortex-M4) と MCU 系列が異なる**。HAL ドライバ (`STM32G0xx_HAL_Driver` vs `STM32L4xx_HAL_Driver`)、ピン配置、ペリフェラル互換性すべて別物なので、片側のコードをそのまま他方に持ち込めない。
- 電源構成:
  - `VIN` (5V) は**本体 CN4 からケーブル経由で供給**される。プローブにバッテリーは無く、単独動作不可。
  - `VIN` → Q1 (**RQ5A030AP** P-MOSFET、電源逆接保護) → `+05P` レール
  - `+05P` → **MAX8881EUT33-T (IC14)** → `+3.3V` → FB3 フェライト → `+3.3VA` (アナログ 3.3V)
  - `+05P` → **LM27761DSGT (IC15)** インバーティング・チャージポンプ → `+05A` / `-05A` (アナログ ±5V)
  - **LM336BMX-2.5 (IC9)** + 2SC2712 (IC8) → `+2.5V` 基準、R39/R40 分圧 + C49 で `1V_REF` 生成
  - **本体と違い LTC3221 は非搭載**。
- センサーフロントエンド（アナログ計測系）:
  - オペアンプ: **TSX562IST × 7** (`IC1`〜`IC7`)。低オフセット・CMOS 入力のレール to レール OP アンプ。
  - アナログマルチプレクサ: **TMUX4053BQBR × 2** (`IC10`, `IC11`)。3回路2:1 ch のアナログスイッチで、`SEL` / `SEL3` ネット (STM32 `IC20.13` / `IC20.14`) でチャネル切替。
  - 外部 ADC: **MCP3424T-E/SL (IC19)** — 4ch 18-bit ΔΣ ADC、I2C1 接続 (STM32 `IC20.1=SDA` / `IC20.32=SCL`)。**STM32 内蔵 ADC は計測系では使っていない**。
  - 入力保護: IC19 の各差動入力に 1N4148WS (`D4`/`D5`/`D6`/`D7`/`D8`/`D9`/`D10`/`D11`) でクランプ。
  - 主要信号系ネット: `MLSS_JUKO` / `MLSS_REF` / `TRA_JUKO` / `TRA_REF`（回路図ブロック単位の信号名）。
- 本体⇔プローブ通信 (RS-232C):
  - STM32 **USART2** (IC20.9=TX=PA2 / IC20.10=RX=PA3) → **MAX3232 (IC21)** → `RX`/`TX` 信号がパッド (TP27=RX, TP28=TX) へ
  - NET: `USART2_TX IC20.9`, `USART2_RX IC20.10`, `RX IC21.13/TP27.1`, `TX IC21.14/TP28.1`
  - **プローブ側にも MAX3232 がある** → ケーブル伝送部は両端とも RS-232 ±12V level（詳細は §7 用語）。
- スイッチ: `S1 (SKRPABE010)` のみ。`NRST2` ラインに接続されたリセットスイッチで、操作ボタン/電源ボタンは無い。
- 外部コネクタ / ケーブル接続:
  - **J4 (5P-SZN, 5pin)**: SWD デバッグ専用（回路図ラベル "ST_Link"。+3.3V / SWCLK / GND / SWDIO / NRST2）。本体 CN1 とは別形状・別ピンアサイン。
  - **本体ケーブルは専用コネクタ無し**。+5V (`TP25` / `VIN`)、GND、RX (`TP27`)、TX (`TP28`) の各パッドにケーブル芯線を直接半田付けする構造（回路図上にも「※相手のRx/Txと接続」と注記あり）。したがってケーブル交換 = プローブ基板ごとの交換 or 再半田付け。

### 共通 / 注意点

- **MCU 系列が異なる**: 本体 = STM32L4 (Cortex-M4), プローブ = STM32G0 (Cortex-M0+)。HAL ドライバ、ペリフェラル、ピン配置はどれも互換性無し。
- **本体⇔プローブ UART は ch / ピン共に別物**:
  - 本体側: STM32L452 **LPUART1** (U4.8/U4.9)
  - プローブ側: STM32G070 **USART2** (IC20.9/IC20.10 = PA2/PA3)
- **RS-232 ケーブル区間は両端 MAX3232 通過後の ±12V level**。両 MCU の UART 〜 各基板 MAX3232 手前までが TTL 3.3V。
- **SWD/JTAG コネクタ形状が違う**: 本体 CN1 (CON8, 8pin) / プローブ J4 (5P-SZN, 5pin)。ケーブル・アダプタを流用不可。
- プローブは本体からのみ給電される（バッテリー非搭載）。プローブ単独でのデバッグ時は外部から +5V 供給が必要。

## 6. ビルド・テストコマンド

両サブプロジェクトとも STM32CubeMX 生成の Makefile ベース。ツールチェーンは `arm-none-eabi-gcc`（`arm-none-eabi-` プレフィクス）で、成果物は各リポの `build/` 配下に `.elf` / `.hex` / `.bin` が生成される。

### 本体 (IM-110, STM32L452RET6 / Cortex-M4)

```bash
# ビルド（並列ビルド可）
cd ../IM-110 && make -j

# クリーン
cd ../IM-110 && make clean
```

- 生成物: `../IM-110/build/IM-110.{elf,hex,bin,map}`
- デフォルトは `DEBUG=1 OPT=-Og`。リリースビルドが必要な場合は `make DEBUG=0 OPT=-Os` のように上書き。

### プローブ (IM-110_Probe, STM32G070 / Cortex-M0+)

```bash
# ビルド
cd ../IM-110_Probe && make -j

# クリーン
cd ../IM-110_Probe && make clean
```

- 生成物: `../IM-110_Probe/build/IM-110_Probe.{elf,hex,bin,map}`
- デフォルトは `DEBUG=1 OPT=-Og`。

### 書き込み・統合テスト

- Makefile には flash ターゲットは含まれていない。STM32CubeProgrammer もしくは ST-Link / JTAG (各基板 CN1 CON8) 経由で `build/*.hex` を書き込む。
- 統合テスト（本体×プローブの RS-232C 通信確認）は現状手順未整備。TODO: JTAG デバッグ環境とログ取得フローを記述。

## 7. 用語

- **RS-232C**: 本資料では本体⇔プローブ間シリアル通信の総称として使う。物理レベルはケーブル伝送区間が両端 MAX3232 通過後の ±12V level、両 MCU の UART ピン〜各基板 MAX3232 手前までが TTL 3.3V（本体 = MAX3232 U16、プローブ = MAX3232 IC21）。
- **インボックス方式**: Claude.ai でのデバッグ記録を `docs/debug-log/` にコピペで蓄積し、Claude Code が後で本体仕様書に統合する運用のこと。
- **真実源 (single source of truth)**: 同じ情報が複数箇所にあると矛盾の原因になるため、プロトコル仕様は `docs/protocol-rs232c.md` 一箇所のみに記述する方針。
