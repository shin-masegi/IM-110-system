## 3. ★ サーマル所見（本実験の最重要知見）

- RX 6900XT 単機で **24B モデルを連続推論**すると発熱が大きく、**サーマルスロットリングが発生**。gen/review が GPU を休みなく回すため冷える隙がない。
- ファン制御が正常なとき、小プロンプトの長文デコードは **~26 tok/s**（本来性能）。ファン制御不調時は長文生成が明確に遅延（実効で概ね半減レベル）。
- **クールダウン挿入**（節間 25〜45秒 idle、モデルは VRAM 常駐のまま演算部のみ休止）を `gen.py` に実装（`SLEEP_BETWEEN`）。総時間は延びるが発熱を抑制できる。
- ファン/温度の直接診断は **Mac 側からは不可**（Ollama API しか見えない）。**サーバー側で `rocm-smi --showtemp --showfan` / `sensors` / `pwm1_enable` の確認が必要**。→ サーバー機上の Claude Code へ委譲。
- **結論（サーバーサイジングへの含意）**: 24B クラスの連続推論では **冷却設計が実効スループットを直接左右する**。単GPUのコンシューマ機（6900XT）を常時ワークロードに使うなら、ファンカーブ見直し・エアフロー確保・（必要なら）クールダウン制御が前提。

## 4. モデル品質と要校正率（2モデル比較）

### 4.1 Devstral（生成）
- **日本語技術文の起稿は良好**。ただし**長文生成で英語へドリフト**する事故が多発（シナリオ・校正手順など）。
- 対策として system プロンプトに **「出力は日本語のみ・英文切替禁止（型番/コマンド名等の固有名詞のみ許可）」** を追加 → **ドリフトはほぼ解消**。厳格プロンプトが有効。
- 誤字（例: 「補正値」→「補正直」）や語の取り違え（「MLSS（ミリリットル/秒）」等）が散発。

### 4.2 mistralprism（レビュー）
- 日本語ドキュメントに対しては **net-negative**。レビュー通過後に **英文へ切替**、一部節を **空欄化(A02)** ・ **大幅短縮(A14, A21)**、指示文のエコー混入などの劣化を招いた。
- → 本番では **日本語ドラフトへの LLM 相互レビューは採用せず**、Claude の Phase 2 校正を検証層とする判断。

### 4.3 事実接地の効果（最大の分岐点）
| 区分 | 与えた事実 | 生成品質 | 要校正 |
|---|---|---|---|
| **プローブ(B)** | 詳細な事実パック（protocol §3 コマンド表・MD 応答・回路） | **高忠実**。コマンド表・MD 応答をほぼ正確に再現 | 軽微（誤字・語取違えのみ） |
| **本体(A)画面** | 事実薄 + ID-200T(DOメーター)テンプレ全文 | テンプレの **DO メーター仕様を大量流用**（`O2_in`・淡水/海水・水温5/20/35℃・溶存酸素・NTU・0.5MHz 等）＋ UI 細部（単位/レンジ/タッチ風ボタン）を**創作** | **重い**。DO 由来記述の削除、単位/レンジの `※要確認` 化、3スイッチ機との整合が必要 |

**要点**: LLM 文書生成の品質は **「与えた事実の質」にほぼ比例**する。事実を厳密に与えた B は即戦力、テンプレ流用に頼った A はハルシネーション温床。**画面 UI 仕様のような一次情報（操作表示一覧・FW）は事実パックに構造化して与えるべき**だった（次回改善点）。

# v2 実験（2026-07-19 未明）: 「Devstral にコードを読ませる」方式

## 動機
v1 は ID-200T(DOメーター)テンプレ流用で**ハルシネーション温床・内容が薄い**（308行）。「事実の質が品質を決める」の教訓を、**"事実＝生の本体FWソース" を渡し、抽出自体を Devstral の仕事にする**と読み替えた。Claude の役割は **マネージャ（ソースのルーティング）＋検証** に限定し、内容は書かない。

## 方式（`tools/spec-gen/v2/`）
1. `slice_funcs.py`: `IM-110/Core/Src/Display.c`(4432行) を **83 の画面描画関数**に切片化。
2. `build_toc_v2.py`: 83 関数を **43 節の粒度 ToC**（ID-200T 相当）へマッピング（＝Claude の管理判断）。
3. `gen_v2.py`: 各節で **関数のソースコードそのもの**＋前提事実＋「ID-200T 相当の粒度で、表示要素は `LS027_disp_*` 呼出から読め、遷移は呼出側にあり不明なら (要確認) と明記」指示を Devstral に投げる。**45秒クールダウン**・**失敗解析ログ完備**（`gen_v2_log.jsonl`＋`prompt_S*.txt`）。
4. `assemble_v2.py`: 本体(A) v2 ＋ プローブ(B) は v1 の忠実版を流用して単一 md 化。

## 結果
| 指標 | v1 | **v2** |
|---|---|---|
| 本体ドキュメント行数 | 308 | **2,409**（≒ ID-200T 級） |
| 成否 | 英語ドリフト・空欄多発 | **43/43 成功・0 失敗** |
| 生成時間 | — | 47 分（+45秒×42 クールダウン）、平均 66s・843tok/節 |
| 事実精度 | DO 由来を大量流用（NTU・溶存酸素・淡水/海水を"該当せず"と誤判定） | **実装から抽出**：透視度=**cm**、SS=mg/L、≥10000→**g/L**、界面=m、界面判断バー118.75%、保存状態(MEM/SAVE/COMPLETE/RELEASE)、淡水/海水(`tansui_sw_flag`)は**実在**と確認 |

**検証(QA)例**: `disp_MLSS_Meas_4`（23行）の実装（`tansui_sw_flag` で海水/淡水タイトル切替・電池・日時・測定モード切替メッセージ・LCD初期化NG処理）を、v2 出力は**過不足なく忠実に記述**。座標や引数(`batvol`/`flash_flg`)の効きもコードから正しく抽出。

## v2 の限界（記録）
- **画面遷移・スイッチ処理は `Display.c`(描画)に無い**（呼出側 `Normal.c`/`Setting.c`/`Adjust.c`）。本ラウンドは未解析で `(要確認: 呼出側FW)` と明示。→ **v3**: 呼出側ファイルも各節の文脈に含めれば遷移まで書ける見込み。
- 英語トークンの微少混入（例: "decision"）が数節。実害は小。
- モデルが節内に独自見出しを付ける癖 → assemble 側で `###` に降格して整形（内容には不干渉）。

## 結論
**オンプレ 24B（Devstral）は「コード→仕様書」の実務を担える。** 鍵は事実の与え方——**要約せず生ソースを渡し、抽出をモデルにさせる**こと。Claude は咀嚼せずルーティングと検証に徹することで、Devstral が主役の意味ある実験になった。

# v3 実験（2026-07-19 午前）: 遷移＋補正式＋EEPROM＋増分再生成、そして VRAM 律速の発見

## 方式（`tools/spec-gen/v3/`）
v2（描画コードのみ）に対し、**呼出側の状態遷移**まで読ませた:
- `caller_context.py`: `Normal.c`/`Setting.c`/`Adjust.c` の 602 `case` ブロックを走査 → 各 `disp_*` に**遷移・スイッチ処理の case ブロックを紐付け**（79/83 関数）。`AllDef.h`（`operation_mode` 状態辞書）も投入。
- `gen_v3.py`: 各節 = 描画コード＋呼出側 case＋状態辞書 → Devstral が**表示要素＋状態遷移＋MEM/DISP スイッチ操作**を執筆。**である調に統一**。付録 A（`Calc.c`＋補正式 xlsx）・付録 B（`Eeprom.h` 割付表）を追加。プローブ B は日本語のみで再生成（英語ドリフト解消）。
- **★ 増分再生成**: 各節の入力ソースを SHA256 ハッシュ化。`--changed-only` で**コードが変わった節だけ再生成**（自己テスト: `disp_STRTDSP.c` 編集 → S01 のみ再生成、で合格）。

## 結果（v1→v2→v3）
| | v1 | v2 | **v3** |
|---|---|---|---|
| 行数 | 308 | 2,409 | **2,451** |
| 画面挙動 | テンプレ流用(幻覚) | 描画から正確抽出 | **＋状態遷移・スイッチ操作**（例: MSR_2→MSR_1） |
| 補正式/EEPROM | なし | なし | **付録A 計算・補正式／付録B EEPROM割付表** |
| 文体 | です/ます混在 | 一部混在 | **である調に統一** |
| プローブ英語 | 混入 | 混入(旧生成) | **解消（0行）** |
| 成否 | 劣化多 | 43/43 | **45/45・0 fail** |

## ★ 最重要ハード知見: 6900XT(16GB) は 24B で VRAM 律速（熱ではない）
v3 序盤で ~8.6 tok/s に低下。当初「サーマルスロットリング」と誤診したが、**server 側での「ジャンクション温度が低い」報告が正しかった**。真因は `/api/ps` で確定:

| num_ctx | 必要メモリ | VRAM/CPU | 実効 decode |
|---|---|---|---|
| 40960（過大設定） | 18.35 GB | VRAM 85% / **CPU 2.67GB 溢れ** | **8.6 tok/s** |
| **12288（適正）** | 収まる | **VRAM 100% / CPU 0** | **22.9 tok/s** |

- **KV キャッシュ（num_ctx）を欲張るとモデルが CPU に溢れ、GPU が遊ぶ → 温度が上がらない → 実効 2.7 倍遅い**。低いジャンクション温度は「throttle」ではなく「GPU が待たされている」サインだった。
- 教訓: **num_ctx は最大プロンプト＋生成に必要な分だけ**（本件は最大 ~6.7k tok → 12288 で十分）。`/api/ps` の `size_vram/size` で 100% GPU を確認するのを運用チェックに含める。
- decode は帯域律速（7900XTX が 6900XT の約2倍＝帯域 1.875 倍と一致）。サーバー選定は「モデルが VRAM に丸ごと載るか」＋「帯域」で決まる。大容量統合メモリ機（Ryzen AI MAX 395 128GB）＝大モデル/長文脈を夜間に、eGPU（R9700 32GB 等）＝高速 tier、の段階導入が現実解。

## 運用フロー（確立）
`SLEEP_BETWEEN=0 NUM_CTX=12288 bash tools/spec-gen/v3/run_v3.sh --changed-only`
→ 再スライス→**変更節のみ** Devstral 再生成→プローブ再生成→結合。コード修正後の仕様書追従が数分で回る。
