# ハードウェア真実源ルール（旧 auto-memory: feedback）

> 出所: Claude auto-memory `feedback_hardware_truth_source.md`（type: feedback / originSession 2b592013…）を 2026-07-22 にリポへ移設。
> メモリ側は本ファイルへのポインタのみ。編集はここを真実源とする。
> 関連の恒久ルールは `CLAUDE.md` §3.5 / §3.5.1 にも明文化済み。

ハードウェア事実（MCU 型番、ピン配置、ペリフェラル割当、コネクタ構成、回路接続）を断定する前に、**判断の根拠を一文で言語化する**こと。優先順位は 回路図 PDF + ネットリスト (`.NET`) > 既存仕様書 > FW 設定 (`.ioc` / `main.c` / Makefile)。

**Why:** 2026-04-22 の IM-110 プロジェクトで、FW の Makefile / `.ioc` を根拠に「プローブ MCU は STM32G070」と断定して `CLAUDE.md` を書き換えたところ、ユーザーから「まて。今ピン配置が正しい/正しくないの判断は何でしてる？プロジェクト運用ルールに記載はないか？できれば回路図＋ネットリストをベースにしたい。今回のように仕様書が間違ってると方向性がおかしくなるので」と方法論そのものを指摘された。既存仕様書 (`CLAUDE.md`) も Claude.ai 自動生成で不正確、FW も人手 CubeMX 設定で基板実態とずれる可能性があるため、どちらも二次資料として扱い、一次資料（schematic + NET）を真実源とする。IM-110 プロジェクトでは `CLAUDE.md` §3.5 / §3.5.1 に明文化した。

**How to apply:**
- 基板系の発言前に `Materials/` 配下の回路図 PDF と `.NET` を開いて根拠確認する。可能なら NET のネット名・コンポーネント ID を併記
- FW 設定と食い違えば「FW が誤設定されている可能性」を疑う順序で処理し、勝手に FW 側を正として書かない
- 一次資料が揃っていない側（例: Materials/ に回路図/NET が不在）については確信を持った更新をせず、ユーザーに資料を要求する
- 別プロジェクトでも同種のハードウェア文書作業では最初に「真実源は何か」「それは現物とリビジョンが揃っているか」を確認するクセをつける
