## 1. 物理層

本体と プローブの UART は別チャネル。両端とも MAX3232 でレベル変換を行い、ケーブル区間は RS-232 ±12V level で伝送する。両端 MCU の UART ピン〜各基板 MAX3232 手前までは TTL 3.3V。

### 1.1 UART パラメータ

| 項目 | 本体 (IM-110T 本体基板) | プローブ (IM-110T プローブ基板) |
|---|---|---|
| MCU ペリフェラル | **LPUART1** | **USART2** |
| MCU ピン (TX / RX) | PC1 / PC0 (U4.9 / U4.8) | PA2 / PA3 (IC20.9 / IC20.10) |
| レベル変換 IC | MAX3232CPWR (U16) | MAX3232CPWR (IC21) |
| ボーレート | **9600 bps** | **9600 bps** |
| データ長 | 8 bit | 8 bit |
| パリティ | なし | なし |
| ストップビット | 1 bit | 1 bit |
| フロー制御 | なし (`UART_HWCONTROL_NONE`) | なし (`UART_HWCONTROL_NONE`) |
| オーバーサンプリング | (LPUART は 16x 固定) | 16x (`UART_OVERSAMPLING_16`) |

根拠:
- 本体: `../IM-110/Core/Src/main.c` 内 `LPUART1_Init` (`hlpuart1.Init.BaudRate = 9600` 等)
- プローブ: `../IM-110_Probe/Core/Src/main.c` 内 `USART2_Init` (`huart2.Init.BaudRate = 9600` 等)
- 本体 `PA11_CTS` / `PA12_RTS` は **外部デバッグコネクタ CN2 (CON6) に引き出されており、プローブ通信には使用していない** (`NET: PA11_CTS U4.44/R16/CN2.3`, `PA12_RTS U4.45/R15/CN2.2`)。

### 1.2 信号経路 / 物理コネクタ

```
[本体側 IM-110T 本体基板]                        [プローブ側 IM-110T プローブ基板]

  STM32L452 LPUART1                                STM32G070 USART2
  PC1=TX (U4.9)  PC0=RX (U4.8)                     PA2=TX (IC20.9)  PA3=RX (IC20.10)
     │   │   (TTL 3.3V)                               │   │   (TTL 3.3V)
     ▼   ▼                                            ▼   ▼
    MAX3232CPWR (U16)                                MAX3232CPWR (IC21)
     │   │   (RS-232 ±12V)                            │   │   (RS-232 ±12V)
     ▼   ▼                                            ▼   ▼
  CN4 (SM05B-SHLS-TF 5pin)  ── 4芯 RS-232 ケーブル ── 基板パッド直結 (半田固定)
  1: +5V        ────────────────────────────────────  TP25 VIN
  2: GND        ────────────────────────────────────  GND パッド
  3: PROBE_RXD  ─── (本体受信 ⇔ プローブ送信) ────────  TP28 TX (IC21.14)
  4: PROBE_TXD  ─── (本体送信 ⇔ プローブ受信) ────────  TP27 RX (IC21.13)
  5: GND        ────────────────────────────────────  GND パッド（予備）
```

- **本体側コネクタ**: CN4 (SM05B-SHLS-TF, JST SH シリーズ 5pin)。基板上の独立コネクタ。
- **プローブ側コネクタ**: **専用コネクタ無し**。回路図 (`回路図_IM-110T_プローブ基板_20260310.pdf`) の注釈「※相手のRx/Txと接続」で明示される通り、ケーブル芯線を TP25 / GND パッド / TP27 / TP28 に直接半田付けする固定構造。したがって **ケーブル単独での交換は不可**、ケーブル交換はプローブ基板ごと交換または再半田付けになる。
- **結線方向**: 本体 PROBE_RXD (= 本体が受信) ⇔ プローブ TX、本体 PROBE_TXD (= 本体が送信) ⇔ プローブ RX。クロス接続を守ること。
- **電源線**: 本体側 CN4.1 (+5V) からプローブへ給電、プローブ側 Q1 (RQ5A030AP, P-MOSFET) で逆接保護後に `+05P` レールへ。プローブ側にバッテリーは無いのでケーブル抜去 = プローブ電源断。

## 2. フレーム構造

本プロトコルは **ASCII テキスト・行指向**。バイナリフレーム (STX/LEN/CRC/ETX 等) や固定長ヘッダは一切無い。

### 2.1 基本形

```
<CMD>[,<param1>[,<param2>[...]]]<CRLF>
```

- `<CMD>`: 2〜4 文字の ASCII コマンド略号（§3 参照）。大文字小文字は区別する（実装は大文字前提）。
- `<param>`: カンマ区切りで 0 個以上。整数 / 浮動小数 / 固定長ビット文字列のいずれか（コマンドにより異なる、§3 参照）。
- `<CRLF>`: `\r\n` (0x0D 0x0A)。送信側は常にこの 2 バイトを付ける。

### 2.2 行終端の取り扱い

- **送信側**: 常に `\r\n` を付与する。プローブ応答 (`IM_110.c:392` 他)・本体送信 (`LinkSerial.c:744` 他) ともに確認済み。
- **受信側**: **LF (`0x0A`) のみで行終端と判定**。CR (`0x0D`) は単なるデータバイトとして受信バッファに積まれるが、行解析時は無視される。
  - プローブ: `IM_110.c:672` `if (RxData == LF) { end = 1; }`
  - 本体: `IM_110.c:195` `if (rx == LF) { Probe_uart_end = 1; }`
- **結論**: LF のみ送る実装とも相互運用可能だが、**両側 FW は CRLF で揃えているのでこれを正とする**。外部ツール/テスト治具もこれに従うこと。

### 2.3 パラメータ区切り

- 区切り文字は **カンマ `,`**。
- **送信側**はカンマのみで密に詰める (`"SADC,0011\r\n"` 等)。
- **本体側の受信パーサはスペース混入も許容**（`IM_110.c:284` `while (*ptr == ',' || *ptr == ' ') ptr++;`）。プローブ応答が `"%f, %f, ..."` (カンマ + スペース) を含むのはこの緩和に合わせたもの。
- **送信時にスペースを入れることは仕様外**。新規コマンド/応答を設計する際はスペース無しで送ること（将来的な厳格化の余地を残すため）。

### 2.4 文字コードと制限

| 項目 | 値 |
|---|---|
| 文字コード | ASCII (7-bit) のみ。非 ASCII バイトは仕様外 |
| 最大フレーム長 | 510 バイト（LF 含む。プローブ受信バッファ 512 バイト − 安全域 2）  |
| バッファオーバーフロー時 | プローブ側 `IM_110.c:678-681` で **強制的に位置 510 を LF に上書きして処理**（エラー応答は出ない）。Flash パラメータ一覧 (`RPP` 応答) は複数行 (各行独立) で 1 行あたり 512 未満に収まる設計 |
| フレームチェックサム | **無し**。パリティ・CRC ともに未使用 |

**バッファオーバーフロー発生条件の見積もり** (v0.4.0 で確認・現状維持):

正常運用では発生しない。発生しうるのは以下のような異常ケースのみ:

| ケース | 発生可能性 |
|---|---|
| 本体 FW が `\r\n` 付与を忘れて長文を送る | FW バグ。開発中の検出対象 |
| 本体 `LinkSerial.c:259` の **BLE パススルー** (`Probe_SendSerialData(&RecvData[2])`) 経由で、BLE クライアントが LF 無しの 512+ バイトを送り込む | 実質皆無。BLE 通信は CPU 側 / BT モジュール側のアクション駆動で、自発的に大量データが流れる構造になっていない。ほぼ全ての判定処理でタイムアウト構造になっており、暴走状態にならない |
| 信号品質劣化で `\n` (0x0A) がビット化けし LF 判定されない | 極めて低確率。§5.3 の PE/FE/NE/ORE 検出が先に発火する想定 |

発生した場合も強制 LF 挿入により「長い偽コマンド」を 1 回処理 → 大抵 `NG\r\n` 応答で終わる副作用で済む。プローブは壊れず、次の行から通常処理に復帰する。

将来、発生頻度が判明して実害が出るなら以下のいずれかを検討（現時点では不要）:
- バッファ全破棄 + リセット（途中コマンドも失われるが状態がクリーン）
- `NG,OVERFLOW\r\n` の明示返送（A.10 の NG 理由コード化と連動）

### 2.5 応答パターン（種別）

応答はコマンドごとに以下のいずれか:

| 種別 | 形式 | 例 | 使用コマンド |
|---|---|---|---|
| `OK\r\n` | 単純成功 | `OK\r\n` | `SP`, `SEL`, `SS`, `MS`, `SID`, `SADZ`, `SADS`, `SADA`, `SADC` |
| `NG\r\n` | 単純失敗 | `NG\r\n` | 上記コマンドのパラメータ範囲外・未知コマンド（§5 参照） |
| 値応答 1 行 | コマンド固有の値文字列 + `\r\n` | `IM-110 Probe Ver.0.11\r\n` | `VR`, `MD` |
| 複数行応答 | 複数行のテキストを個別 `\r\n` で送信、末尾に `OK\r\n` or 固定文 | (§3.2 `RPP` 参照) | `RPP`, `WPP`, `RPI` |
| 自律送信 | `MS,1` ON 中、プローブから周期送信 | (§3.3 参照) | `MS` (MS_flag ON 時) |

### 2.6 エンディアン

ASCII テキスト前提のためエンディアン概念は適用されない（数値は 10 進文字列 or float 文字列で送受信）。

## 3. コマンド一覧

全コマンドは **本体 → プローブ方向** のみで発行される。プローブから本体への自律送信は §3.3 (MS) のみ。

**大文字小文字**: コマンド名は **大文字 (A-Z) のみ** 有効。小文字・大小文字混在 (`vr`, `Vr`, `mS` 等) はすべて `NG\r\n` 返送される (プローブ実装が `strncmp` で大文字比較するため)。ユーザー入力 / 治具からコマンドを送る際も大文字で統一すること。

### 3.2 応答詳細（値応答・複数行応答）

#### 3.2.1 `VR` — 製品名・バージョン

```
→ VR\r\n
← IM-110 Probe Ver.0.11\r\n
```

- 定義: `IM-110_Probe/Core/Inc/IM_110.h` `#define P_NAME "IM-110 Probe"`, `#define P_VER 11`
- 書式: `"%s Ver.%d.%02d\r\n" → name, ver/100, ver%100` (`IM_110.c:392`)

#### 3.2.2 `MD` — 測定値 1 回送信（および MS 自律送信の中身）

```
→ MD\r\n
← <mV[0]>, <mV[1]>, <mV[2]>, <mV[3]>, <mV[4]>, <pressure hPa>\r\n
```

- 6 個の `float` をカンマ + スペースで区切り（`"%f, %f, %f, %f, %f, %f\r\n"`、プローブ `IM_110.c:405`）
- 内部配列 `AD_AVE_mV[5]` (float) + 気圧センサ値 1 個
- 移動平均は `SADA` で設定した回数で計算済み
- pressure hPa: HSPPAD143A 気圧センサーの値。**現行リビジョン (IM-110T プローブ基板 2026-03-10) ではフットプリント未実装のため常に 0 前後**（回路図注記「未実装」参照）。FW 側には `HSPPAD143_ReadPressure()` 関連コードが既に実装されており、次回プローブ基板改版で HSPPAD143A (I2C2 / PA11/PA12) が実装されたら値が有効化する。**`MD` 応答書式は変更しない** — 6 番目の float が未実装時 0 → 実装後に実測 hPa に切り替わるだけで、本体パーサは不変
- HSPPAD143A 内蔵の**温度センサ値は `MD` 応答には採用しない** 方針（本水質計測装置の用途上必要な情報は気圧のみ）。将来温度計測が必要になった場合は `MD` 書式を拡張せず、新コマンド（例: `MDT` = Measure Data with Temperature）を別途追加する

**`AD_AVE_mV[0..4]` の各要素が指すもの**:

`AD_AVE_mV` の index は MCP3424 の物理 ch と 1 対 1 ではなく、**`SEL3` (`SS` コマンド) の状態でインデックスが動的に変わる** ため注意が必要:

| 配列 index | 対応する物理計測 | 更新条件 | 用途の想定 |
|---|---|---|---|
| `AD_AVE_mV[0]` | MCP3424 CH1 | `SADC` bit 0 = 1 | — |
| `AD_AVE_mV[1]` | MCP3424 CH2 | `SADC` bit 1 = 1 | — |
| `AD_AVE_mV[2]` | MCP3424 CH3 の **SEL3 = LOW** 時 | `SADC` bit 2 = 1 **かつ** `SS,0` | 通常ゲイン x1 測定 |
| `AD_AVE_mV[3]` | MCP3424 CH4 | `SADC` bit 3 = 1 | — |
| `AD_AVE_mV[4]` | MCP3424 CH3 の **SEL3 = HIGH** 時 | `SADC` bit 2 = 1 **かつ** `SS,1` | 同一 ch3 を PGA gain x2 で再測定 (`IM_110.c:196-201`) |

要点:
- **`SADC` は 4 bit (ch0-ch3) の制御のみ**。`AD_AVE_mV[4]` は SADC で直接有効/無効化できず、「ch2 が有効 + SEL3 HIGH」の組み合わせで自動的に更新される
- `SS,0` と `SS,1` を切り替えると、**同じ物理 ch3 の読み取り値が `[2]` と `[4]` のどちらに流れるかが切り替わる**。更新されなかった側は古い値が残留

**無効化 ch / 未更新 ch の値**:

`AD_AVE_mV[]` はゼロ初期化 (C 標準、グローバル) される。一度も有効化されていない ch は `0.000000`。一度有効化されたが後で無効化された ch は **直前の値が残留** (クリア処理なし)。本体側は必要に応じて `SADC` / `SS` 設定を自身で保持し、有効な index だけを読む責任がある。

**具体例**:
```
起動直後 SADC=0000:        0.000000, 0.000000, 0.000000, 0.000000, 0.000000, 0.00
SADC,0011 (ch0+ch1 有効):  1.234,    -0.567,   0.000000, 0.000000, 0.000000, 0.00
SADC,1111 + SS,0:          1.234,    -0.567,   2.100,    3.456,    0.000000, 0.00
SADC,1111 + SS,1:          1.234,    -0.567,   2.100,    3.456,    5.678,    0.00  (※[2]は据え置き、[4]が更新)
SADC,0000 に戻す:          1.234,    -0.567,   2.100,    3.456,    5.678,    0.00  (全て残留)
```

  | Product Name | `"Product Name: %s\r\n"` | `Product_Name` (文字列) |
  | Probe ID | `"Probe ID: %lu\r\n"` | `Probe_ID` (uint32 を **10 進数**で出力、`SID,N` 入力と同じ base) |
  | ADC Zero[N] | `"ADC Zero[%d]: %f\r\n"` | `ADC_Zero[0..4]` (float、デフォルト精度 `%f` = 小数6桁) |
  | ADC Span[N] | `"ADC Span[%d]: %f\r\n"` | `ADC_Span[0..4]` (現行ユーザー補正後スパン) |
  | LED Out[N] | `"LED Out[%d]: %f\r\n"` | `LED_Out[0..4]` (LED 制御出力値) |
  | ADC Span_S[N] | `"ADC Span_S[%d]: %f\r\n"` | `ADC_Span_S[0..4]` (**出荷時スパン** — 工場校正値、`RPI` でのみ復帰) |
  | OK | `"OK\r\n"` (固定) | — |

- **Probe ID の表記**: 入出力とも **10 進 `uint32`**（`SID,N` 入力 = `strtoul(..., 10)`、`RPP` 出力 = `"%lu"`）。内部保持型は同一の `uint32_t` で、base を揃えたことで目視照合が容易。**v0.4.0 で 16 進 `%08lX` 出力から変更**（プローブ FW 改版で反映）。
- **消費側の解析方針**: 各行を `\r\n` で分割し、コロン `: ` で key / value に分離。最終行の `OK\r\n` で送信完了を判定（途中で `OK` 以外で終わったら切断扱い）。

#### 3.2.4 `WPP` / `RPI` — 固定文字列応答

| コマンド | 成功応答 | 失敗応答 |
|---|---|---|
| `WPP` | `Parameters saved to flash.\r\n` | `Failed to write parameters to flash.\r\n` |
| `RPI` | `Parameters reset to initial values.\r\n` | (失敗パターンなし) |

- `IM_110.c:801-814` 参照

### 3.3 自律送信（`MS` 有効時の周期送信）

- トリガ:
  - **`MS,1\r\n`** (推奨、v0.4.0 以降): `MS_flag = 1` を明示セット。冪等なので重複送信しても ON のまま
  - **`MS,0\r\n`**: `MS_flag = 0` を明示セット。冪等な停止
  - **`MS\r\n`** (引数省略、レガシー): 受信毎に `MS_flag ^= 1` でトグル。プローブ単体試験用途で残置
- ON 時はプローブが `TIMER_UART_OUTPUT = 1000 ms` (`IM_110.h:87`) 周期で §3.2.2 の `MD` 応答と同一フォーマットの測定値行を送信
- 実装: `main.c:149-154` (プローブ側、TIM1 1ms 割り込みでソフトタイマを進めて周期判定)
- 自律送信中にコマンド応答（`OK` など）が同じチャンネル上で混在する可能性があるが、本体パーサは §3.4 の行分類ルールで確実に区別できる。

### 3.4 応答行と測定値行の区別

MS 自律送信が ON の状態でコマンドを発行すると、測定値行と応答行が同じ UART ストリームに流れる。本体パーサは **受信した 1 行の先頭 1 文字** で用途を判別する。

**分類ルール**:

| 先頭 1 文字 | 用途 | 処理 |
|---|---|---|
| `0`–`9` または `-` | 測定値行（`MD` / 自律送信） | `strtof` で 6 個の float をパース、測定値として処理 |
| `A`–`Z`（大文字アルファベット） | コマンド応答 | 直前に送ったコマンドの応答として扱い、§3.2 の書式で解釈 |
| 空行 / `\r` 単独 / その他制御文字 | ノイズ | 無視 |

**網羅性の根拠**（プローブが実際に送る行の先頭文字一覧）:

| 行の種別 | 先頭文字 | 例 |
|---|---|---|
| `MD` / MS 自律送信の測定値 | `0`–`9` / `-` | `1.234, -0.567, 2.100, 0.000, 0.000, 0.000\r\n` |
| `OK` 応答 | `O` | `OK\r\n` |
| `NG` 応答 | `N` | `NG\r\n` |
| `VR` 応答 / 起動バナー | `I` | `IM-110 Probe Ver.0.11\r\n` |
| `RPP` Product Name / Probe ID / RPI / WPP 成功 | `P` | `Product Name: IM-110\r\n`, `Probe ID: 1234567890\r\n`, `Parameters saved to flash.\r\n`, `Parameters reset to initial values.\r\n` |
| `RPP` ADC Zero/Span/Span_S | `A` | `ADC Zero[0]: 0.000000\r\n` |
| `RPP` LED Out | `L` | `LED Out[0]: 0.458000\r\n` |
| `WPP` 失敗 | `F` | `Failed to write parameters to flash.\r\n` |

→ 測定値行は必ず `0-9` または `-` で始まり、応答行は必ずアルファベットで始まる。**両者は先頭 1 文字で排他的に区別可能**。

**プローブ側の契約（逐次保証）**:
- プローブは 1 つのコマンドに対する応答を **完結させてから MS 自律送信を再開する**。単一応答の途中（例: `RPP` の複数行ダンプの途中）に測定値行が割り込むことは無い
- 現状実装: プローブ main loop は `CheckSerialCMD()` の処理と `uart_Put_Measure_Data()` (自律送信) を逐次で呼んでおり、それぞれのブロック内で送信を完結させるためインターリーブは発生しない (`main.c:137-154`)
- 本仕様はこの逐次性をプローブ側 FW の契約として固定する。将来プローブを RTOS 化する等して送信処理を並行化する場合、この契約を破らないよう送信のクリティカルセクションで排他制御すること

**本体パーサ状態**:
- 本体は応答待ちフラグ (`Probe_cmd_pending`, `IM_110.c:121`) を持つが、**測定値行の受信は応答待ちの有無にかかわらず常時受け付ける**
- 応答行が到着した場合は到着順で `cmd_pending` に対応付けてタイムアウトタイマー (§4.3) を止める
- 応答待ちフラグが立っていない状態で応答行が来た場合（通常発生しないが、起動時バナー等）はログのみ記録して破棄

## 4. タイミング

### 4.1 物理・HAL 層

| 項目 | 値 | 備考 |
|---|---|---|
| ボーレート | 9600 bps | 1 バイト ≒ 1.04 ms (10 bit frame) |
| 典型コマンド長 | 〜20 バイト | 1 行送信 ≒ 20 ms |
| `MD` 応答長 | 〜80 バイト (float 6 個) | 1 行送信 ≒ 80 ms |
| `MS` 自律送信周期 | **1000 ms** | `TIMER_UART_OUTPUT` (プローブ `IM_110.h:87`) |
| 本体 `HAL_UART_Transmit` タイムアウト | 100 ms | `IM-110/Core/Src/IM_110.c:169`（送信側ブロック時間） |
| プローブ起動完了待ち | 暗黙（電源立ち上がり数十 ms） | 本体は PROBE 電源供給後すぐに `MS\r\n` を送る。明示的な待機は無し |

### 4.2 コマンド応答タイムアウト（本体側）

本体がコマンドを送信してから、プローブ応答行（`OK` / `NG` / 値応答 / 複数行の先頭）を 1 行以上受信するまでの上限。**コマンド種別により 2 値**:

| 区分 | タイムアウト値 | 対象コマンド |
|---|---|---|
| **fast** | **500 ms** | `VR`, `SP`, `SEL`, `SS`, `MS`, `MD`, `SID`, `SADZ`, `SADS`, `SADA`, `SADC`, `RPI` |
| **slow** | **3000 ms** | `RPP`（23 行ダンプ、約 720 ms 伝送 + 処理）、`WPP`（Flash 書込 100-500 ms を含む） |

- FW 側で `#define CMD_TIMEOUT_FAST_MS 500` / `#define CMD_TIMEOUT_SLOW_MS 3000` 等として定数化する
- コマンド発行時、ディスパッチャは区分を参照して待機タイムアウトを選ぶ
- `RPP` / `WPP` 以外の新コマンドを追加する場合はデフォルト fast とみなし、実測で slow が必要な場合のみ分類を見直す
- MS 自律送信中のコマンド応答は測定値行と混在する。本体パーサは §3.3 の規則（行頭が数字 or `-` なら測定値、非数字なら応答）で分岐する

### 4.3 タイムアウト時のリトライ

**コマンド応答タイムアウト → 原則 1 回リトライ**。ただし以下のコマンドはリトライしない（副作用回避のため）:

| コマンド | 除外理由 |
|---|---|
| `SADZ,N` | 現在の AD 値をゼロ基準として記録する。時間経過で AD 値がドリフトするため、リトライ時の校正結果が 1 回目と異なる値になる |
| `SADS,N` | 同上（スパン基準の上書き） |
| `MS` (引数省略形) | プローブ実装がトグル式 (`MS_flag ^= 1`) のため、2 回実行すると元の状態に戻る (OFF→ON→OFF)。ただし新規本体コードは **`MS,1` / `MS,0` の明示指定を使う**（冪等、§3.3 推奨）ためリトライ可能 |

リトライしないケースでタイムアウトが起きた場合は即座に失敗扱い。UI には積極的エラー表示せず、ログのみ記録（§5.5 参照）。

**MS 自律送信（測定値行）の受信タイムアウト → リトライしない**:

- 上限は **2000 ms**（自律送信周期 1000 ms の 2 倍、ジッタ吸収のため）
- 該当測定値の UI 表示を `----` に切り替える
- 次の有効測定値行が届いたら自動で通常表示に復帰
- 連続的に測定値タイムアウトが続く場合は §5.4 の切断判定に移行する（具体ロジックは §9 TODO）

## 5. エラー処理

### 5.1 プローブ側の応答エラー (`NG\r\n`)

以下のケースでプローブは `NG\r\n` を返す:

| 発生条件 | 対象コマンド | 判定箇所 |
|---|---|---|
| パラメータ欠落（カンマの後に値が無い） | `SP`, `SEL`, `SS`, `SID`, `SADZ`, `SADS`, `SADA`, `SADC` | 各コマンドの `strtok` 後 NULL チェック |
| パラメータ形式不正（`strtof`/`strtol`/`strtoul` の失敗） | 同上 | `endptr` チェック |
| パラメータ範囲外 | `SP` (0.0〜1.0)、`SEL` (0〜2)、`SS` (0〜1)、`SADZ`/`SADS` (0〜4)、`SADA` (0〜255) | 各レンジ比較 |
| `SADC` の書式不正 | 4 文字の `'0'`/`'1'` 以外を含む | `IM_110.c:543-550` |
| 14 コマンドのどれにもマッチしない | 全未知コマンド | `IM_110.c:386` |

`NG\r\n` には理由コードが無いので、本体側は送信直前のコマンドと照合して原因を推定する。

**理由コード化しない方針** (v0.4.0 で確定): 本プロトコルは IM-110 本体⇔プローブの閉じた系であり、本体側の NG 発生時の挙動は §4.3 / §5.5 に従い「リトライ → 失敗なら `----` 表示 / ログ記録」で一律となる。理由（`RANGE` / `FORMAT` / `UNKNOWN` 等）が分かっても本体の分岐が変わらないため、理由コードを返す実装コストに対する価値が薄い。デバッグ用途も本体側で「送信したコマンド文字列 + NG 到来」のログから推定可能。将来、BLE 経由で外部診断治具を本格運用する等の要求が出てきたら再検討する。

### 5.2 CRC / チェックサム

- **RS-232C 通信では CRC/チェックサムを一切使用していない**。ノイズによる文字化けは検出できない。
- （参考）本体側 1-Wire EEPROM 通信 (`Wire.c`) には XOR チェックサムがあるが、これは RS-232C とは別系統。

### 5.3 HAL 層のエラー処理

**仕様**: 両側とも `HAL_UART_ErrorCallback()` を実装し、パリティ / フレーム / ノイズ / オーバーラン (PE/FE/NE/ORE) フラグをクリアした上で DMA 受信を再開する。上位層への通知は行わない（§5.5 の積極的エラー表示なしポリシーに準拠）。エラーカウンタは現時点では保持しない。

- **本体**: 実装済み (`IM-110/Core/Src/LinkSerial.c:200-212`)。PE/FE/NE/ORE フラグクリア後に DMA 受信を再開
- **プローブ**: **現状未実装、v0.4.0 で仕様化、次回 FW 改版で追加**。本体 `LinkSerial.c:200-212` と同じロジックを `USART2` / `huart2` 向けに移植する。STM32G0xx HAL API は STM32L4xx とほぼ同一なので流用可能。実装イメージ:

  ```c
  void HAL_UART_ErrorCallback(UART_HandleTypeDef *huart) {
      if (huart->Instance != USART2) return;
      __HAL_UART_CLEAR_PEFLAG(huart);  // PE/FE/NE/ORE 一括クリア
      HAL_UART_Receive_DMA(huart, RecvData, sizeof(RecvData));
  }
  ```

プローブ側未実装による現実リスクは限定的（プローブが受ける最長コマンドは `SID,4294967295\r\n` = 15 バイト、受信バッファは 512 B でオーバーフロー余地は十分）だが、電源ノイズや信号品質劣化で FE / NE が発生した場合に DMA が停止したまま復旧しない懸念が残るので、次回改版で確実に入れる。

### 5.4 通信切断の判定

**方式: MS 自律送信の間隔監視（パッシブ監視）** — 定期 VR ヘルスチェック等のアクティブ監視は採らない。

**判定ロジック**:

1. 本体は MS 自律送信の測定値行が届くタイミングを監視する
2. 直前の測定値行から **2000 ms** 経過しても次の行が届かなければ「1 回タイムアウト」とカウント（§4.3 の MS 受信タイムアウト）
3. **連続 3 回タイムアウト（= 6 秒無応答）** で切断扱いに遷移する
4. 切断扱い中に測定値行が 1 行でも届いた時点で、自動的に接続扱いへ復帰する（リセット時復帰処理は不要）

**前提**:
- この方式は `MS` 自律送信が ON の状態を前提とする。本体起動シーケンス (§6) で `MS\r\n` を 1 回送るので通常運用では必ず ON
- `MS` OFF 状態（調整モード等、将来の運用で `MS` をトグル OFF する場合）では本方式は機能しない。将来 MS OFF 運用を導入する際は別途アクティブヘルスチェックの追加を検討する（§9 TODO）

**UI の扱い** (§5.5 と統一):
- 切断扱い中の数値表示は引き続き `----`（ロスト扱いと切断扱いで UI 差別化しない）
- 接続状態を明示するアイコン / インジケーターは **現時点では設けない**。将来の設定画面 / 通知 UI 追加時に再考する

**本体 FW 実装の想定**:
- MS 受信ハンドラで失敗カウンタをインクリメント、成功受信時に 0 リセット
- カウンタが 3 以上なら全測定値 UI を `----` 表示のまま据え置き、ログに「probe disconnected」を 1 回記録（切断遷移時のみ、連続で出さない）

### 5.5 UI / 運用ポリシー（タイムアウト時）

- **積極的なエラー表示はしない**: モーダルダイアログ / トースト / 画面上のエラーバナー等は出さない。ユーザー操作を邪魔する表示は避ける
- 失敗（コマンドリトライ後の失敗、測定値未到着、HAL エラー等）は **ログのみ** 記録する（デバッグ用、本番 UI には露出させない）
- MS 自律送信の測定値行が §4.3 に従い 2000 ms 届かない場合は、該当値の UI を `----` に置き換え、次の有効値で自動復帰
- 連続的なタイムアウトが発生した場合は §5.4 の通信切断判定に移行する（判定条件は §9 TODO で別途仕様化）

## 6. 起動シーケンス

### 6.1 現行 FW の実装 (本体 Ver.0.14 × プローブ Ver.0.11、ハンドシェイク無し)

```
[プローブ側]
1. 電源投入 (本体 CN4.1 → ケーブル → プローブ TP25 VIN)
2. STM32G070 起動 → HAL 初期化 → USART2 DMA 受信開始 (main.c:122 uart_init)
3. バナー送信: "IM-110 Probe Program Test\r\n" (main.c:123)
4. Flash パラメータ読込、MCP3424 / HSPPAD143 初期化
5. コマンド受信待ちループへ (`CheckSerialCMD` を LF 受信毎に呼ぶ)

[本体側]
A. 電源投入 → STM32L452 起動 → HAL 初期化
B. LPUART1 DMA 受信開始 (Probe_uart_init, IM_110.c:148-159)
C. プローブ電源供給 (本体→プローブ +5V 給電線、CN4.1)
D. 一定時間経過後、Probe_Request_MD() で "MS\r\n" 送信 (IM_110.c:218-224)
   — ここで `Probe_MS_started` フラグを立てる (1 回限り)
E. プローブから 1 秒周期で測定値が自動送信されるので、Probe_Process_MD() で解析
```

- プローブ起動バナー (`IM-110 Probe Program Test\r\n`) は本体パーサで数字/符号以外の行として無害に捨てられる (§3.3 の区別ルール)
- 疎通確認も Ver 照合も無し。ケーブル抜け等の異常を起動時に検出できない

### 6.2 仕様: 軽量ハンドシェイク (v0.4.0 で規定、本体 FW 実装待ち)

本体起動時に `VR` で疎通確認とログ記録を行う軽量ハンドシェイクを導入する。**Ver 照合は当面行わない**、失敗しても通常動作 (`MS\r\n` 送信) へ進む。将来 §7 の互換性ポリシー有効化や完全なハンドシェイクに段階的に拡張する余地を残す。

```
[本体側の新シーケンス]
A. 電源投入 → STM32L452 起動 → HAL 初期化
B. LPUART1 DMA 受信開始
C. プローブ電源供給
D. 200 ms 待機 (プローブ STM32G070 起動完了を待つ暗黙マージン)
E. "VR\r\n" 送信 → fast タイムアウト 500 ms 待機 (§4.2)
   - 応答が "IM-110 Probe Ver." で始まる → ログ "Probe connected: <応答文字列>"
   - タイムアウト → 1 回リトライ → 失敗ならログ "Probe not responding at startup"
   - 応答が異形式 → ログに原文記録
F. いずれの結果でも "MS,1\r\n" 送信 → 通常動作へ (明示 ON 指定で冪等、§3.3)
G. プローブから 1 秒周期で測定値を受信 (§3.3)
```

**方針のポイント**:
- **E で失敗しても F で MS を送る**: プローブが後から物理接続 / 電源投入された場合でも、MS 自律送信の最初の 1 行で §5.4 の切断判定が復帰扱いに戻る（能動再試行は不要）
- **200 ms 待機の根拠**: プローブ側 HAL 初期化 + Flash 読込 + MCP3424 初期化で数十〜100 ms 程度かかると見積もり、マージンを含めて 200 ms。実機計測で調整の余地あり
- **ログ出力のみ、UI 表示はしない**: §5.5 の積極的エラー表示なしポリシーに統一

### 6.3 将来案: プローブ起動時の MS 自動開始

プローブ FW を次に改版するタイミングで、**起動時 `MS_flag = 1` 相当を内部でセット** する変更を入れる選択肢がある。

- 実装箇所: プローブ `main.c` の初期化ブロックで `MS_flag = 1` を既定値に
- メリット:
  - 本体は `MS` 送信を省略でき、§6.2 のハンドシェイクが純粋に「疎通ログ記録」のみに簡素化
  - 再接続時 (ケーブル抜き差し) も本体介在なしに自動送信が復旧
- デメリット:
  - 調整モードで MS を止めたい場合、明示的に `MS,0` 等で OFF する必要あり（現状実装はトグル式、§9 TODO の `MS` 引数仕様化と連動）
  - プローブ単独で生測定値を流し続けるため、想定外の受信先で問題になる可能性

## 7. プロトコルバージョニング

このプロトコルは `MAJOR.MINOR.PATCH` で管理する。

- **MAJOR**: フレーム構造やコマンド体系の後方互換を破る変更
- **MINOR**: コマンドの追加（既存コマンドは不変）
- **PATCH**: ドキュメント整備、仕様明確化のみでバイナリ不変

### 7.1 互換ポリシー

本体とプローブの互換性判定ルール:

- MAJOR 一致必須
- 本体の MINOR ≥ プローブの MINOR （＝本体は古いプローブを許容する、逆は不可）
  ※ このポリシーはプロジェクトに合わせて書き換えてください

### 7.2 バージョン管理とリリース

- 本リポジトリ (IM-110-system) のこのドキュメント冒頭の `版` を更新する
- 本体側 (`IM-110`) のコードに `#define PROTOCOL_VERSION_*` を埋め込む
- プローブ側 (`IM-110_Probe`) のコードにも同じ定数を埋め込む
- 両側で `GET_VERSION` 経由で突合可能にする
