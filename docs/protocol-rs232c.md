## 2. フレーム構造（雛形 — 実装から埋めること）

```
+--------+--------+--------+--------+--------+------+--------+
| STX    | LEN    | CMD    | SEQ    | DATA.. | CRC  | ETX    |
| 1 byte | 1 byte | 1 byte | 1 byte | N byte | 2 B  | 1 byte |
+--------+--------+--------+--------+--------+------+--------+
```

> 上記は **仮のフォーマット例** です。実装に合わせて書き換えてください。

| フィールド | サイズ | 説明 |
|---|---|---|
| STX | 1 | フレーム開始 (例: 0x02) |
| LEN | 1 | DATA 部のバイト数（ヘッダ・CRC・ETX を含まない） |
| CMD | 1 | コマンド ID (下表参照) |
| SEQ | 1 | シーケンス番号 (応答と対応付け) |
| DATA | N | コマンド固有ペイロード |
| CRC | 2 | CRC16 (多項式 TODO) |
| ETX | 1 | フレーム終端 (例: 0x03) |

## 6. 起動シーケンス

本体とプローブが電源投入されてから通信を確立するまでの手順。

```
1. 本体: 電源投入 → 3.3V 安定 → MCU 初期化
2. 本体: PROBE 電源供給開始 (PA1_5vON assert)
3. プローブ: 起動完了、初期化完了後に通信ループ開始
4. 本体: PING を送信し応答を確認 (TODO: タイムアウト値)
5. 本体: GET_VERSION でプローブファーム版を取得
6. 本体: バージョン互換性チェック ← §8 参照
7. 通常動作開始
```

## 7. プロトコルバージョニング

このプロトコルは `MAJOR.MINOR.PATCH` で管理する。

- **MAJOR**: フレーム構造やコマンド体系の後方互換を破る変更
- **MINOR**: コマンドの追加（既存コマンドは不変）
- **PATCH**: ドキュメント整備、仕様明確化のみでバイナリ不変

### 7.1 互換ポリシー

本体とプローブの互換性判定ルール:

- MAJOR 一致必須
- 本体の MINOR ≥ プローブの MINOR （＝本体は古いプローブを許容する、逆は不可）
  ※ このポリシーはプロジェクトに合わせて書き換えてください

### 7.2 バージョン管理とリリース

- 本リポジトリ (IM-110-system) のこのドキュメント冒頭の `版` を更新する
- 本体側 (`IM-110`) のコードに `#define PROTOCOL_VERSION_*` を埋め込む
- プローブ側 (`IM-110_Probe`) のコードにも同じ定数を埋め込む
- 両側で `GET_VERSION` 経由で突合可能にする
